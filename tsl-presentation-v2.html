// 00 — fondamenta
Prima di tutto: cosa è uno shader?
Se conosci già GLSL, salta pure alla slide 4. Altrimenti — 3 minuti e hai tutto il contesto.
🖥️ CPU vs GPU

Il tuo JavaScript gira sulla CPU — un processore potente ma con pochi core (4–16). La GPU ha migliaia di core più semplici. Uno shader è un programma che gira su GPU, eseguito in parallelo per ogni pixel o vertice.

📐 Vertex Shader

Viene chiamato una volta per ogni vertice della mesh 3D. Il suo compito: dire dove quel punto appare sullo schermo. Puoi usarlo per animare, deformare, fare onde.

🎨 Fragment Shader

Viene chiamato una volta per ogni pixel da colorare. Il suo compito: restituire un colore. Può leggere texture, calcolare luci, fare effetti visivi. È dove succede tutta la magia visuale.

La pipeline grafica
JavaScript (CPU)
Definisce geometrie, materiali, uniforms. Chiama il renderer ogni frame.
Vertex Shader (GPU)
Per ogni vertice: proietta nello spazio 2D dello schermo. Può deformare posizioni.
Rasterization (automatica)
La GPU converte i triangoli in pixel, interpolando i varyings tra vertici.
Fragment Shader (GPU)
Per ogni pixel: calcola e restituisce il colore finale. Qui vivi gli effetti.
💡 Un varying è un valore calcolato nel vertex shader e interpolato per ogni pixel. Se vertice A = rosso e vertice B = blu, il pixel a metà riceve viola. È così che funzionano i gradienti base.

// 01 — il problema
GLSL in Three.js: il vecchio modo
Prima di TSL, scrivevi shader come stringhe GLSL. Funzionava, ma era fragile, verbose e non componibile.
ShaderMaterial — approccio classico
// ❌ Tutto come stringhe — zero type-checking
const mat = new THREE.ShaderMaterial({
  uniforms: {
    uTime: { value: 0 },
    uColor: { value: new THREE.Color('red') }
  },
  vertexShader: `
    uniform float uTime;
    varying vec2 vUv;
    void main() {
      vUv = uv;
      vec3 pos = position;
      pos.y += sin(pos.x*3.0 + uTime) * 0.2;
      gl_Position = projectionMatrix
        * modelViewMatrix * vec4(pos,1.0);
    }
  `,
  fragmentShader: `
    uniform vec3 uColor;
    varying vec2 vUv;
    void main() {
      gl_FragColor = vec4(uColor*vUv.x, 1.0);
    }
  `
});

// Loop — update manuale degli uniforms
function animate() {
  mat.uniforms.uTime.value = clock.getElapsedTime();
  renderer.render(scene, camera);
  requestAnimationFrame(animate);
}
🚨 Problemi del vecchio approccio
  • Stringhe GLSL — nessun autocomplete, nessun type-check, errori solo a runtime
  • Non componibile — non puoi "importare" funzioni da altri file facilmente
  • WebGL-only — GLSL non funziona su WebGPU che usa WGSL
  • Uniforms manuali — aggiornamento esplicito ogni frame
  • onBeforeCompile — hack per modificare shader built-in, fragile tra versioni
⚠️ onBeforeCompile era il metodo per modificare i materiali standard (MeshStandardMaterial, ecc.) iniettando codice GLSL. Funzionava ma si rompeva ad ogni aggiornamento di Three.js.
✅ Con TSL: JavaScript nativo, type-safe, cross-backend (GLSL + WGSL), aggiornamento uniforms automatico, funzioni riutilizzabili.
// 02 — architettura
Come funziona TSL internamente
1. Node Graph

Ogni operazione TSL crea un Node. Le operazioni si concatenano in un DAG (directed acyclic graph). Non eseguono codice — descrivono una computazione.

2. Code Generation

Il NodeBuilder visita il grafo e genera GLSL (WebGLRenderer) o WGSL (WebGPURenderer). Stessa codebase, due backend.

3. NodeMaterial

I nodi vengono assegnati come proprietà (colorNode, normalNode…) di un NodeMaterial. Il renderer compila al primo uso.

node-graph.js — i nodi sono lazy
import { vec3, sin, add, time } from 'three/tsl';

// ⚠️ Questo NON calcola niente subito!
// Costruisce un grafo di nodi (AST)
const colorNode = vec3(
  sin(add(time, 0.0)),
  sin(add(time, 2.094)),
  sin(add(time, 4.189))
);

// Solo quando assegni al materiale e il renderer
// chiama render(), il grafo viene compilato in GLSL/WGSL
const mat = new MeshBasicNodeMaterial();
mat.colorNode = colorNode;

// GLSL generato automaticamente (circa):
// vec3 color = vec3(sin(time), sin(time+2.094), sin(time+4.189));
⚠️ La distinzione fondamentale: i valori TSL non sono numeri JavaScript. float(1.0) crea un FloatNode, non il numero 1.0. Non puoi fare float(1) + 2 — devi fare float(1).add(2).

// 05 — uniforms & varyings
Uniforms reattivi e Varyings
uniforms.js
import { uniform } from 'three/tsl';

// Crea uniform — aggiorna .value per cambiare
const uColor = uniform(new Color('hotpink'));
const uIntensity = uniform(1.0);
const uOffset = uniform(new Vector2());

// Usalo nel grafo
mat.colorNode = uColor.mul(uIntensity);

// Aggiornamento da JS — nessun rebuild shader!
animate(() => {
  uIntensity.value = Math.sin(Date.now() * .001);
  // uColor.value.set('blue') — funziona anche su Color
});
✅ Cambiare .value aggiorna lo shader al frame successivo senza recompilation. Enorme vantaggio vs GLSL manuale.
varyings.js
import { varying, positionLocal,
  Fn, vec3 } from 'three/tsl';

// Dichiara un varying (interpolato GPU)
const vWorldPos = varying(vec3());

// Vertex: assegna il valore
mat.positionNode = Fn(() => {
  vWorldPos.assign(positionLocal);
  return positionLocal;
})();

// Fragment: usa il valore interpolato per pixel
mat.colorNode = vWorldPos
  .abs().fract(); // colori dalla posizione

// 06 — funzioni
Funzioni shader con Fn()
TSL permette di definire funzioni shader riutilizzabili in JavaScript, con loop, conditional e return.
Fn() — funzioni e control flow
import { Fn, float, vec3, If, Loop } from 'three/tsl';

// fBm — fractal brownian motion noise
const fbm = Fn(([p]) => {
  const value     = float(0).toVar();   // mutabile!
  const amplitude = float(0.5).toVar();
  const pos       = p.toVar();

  Loop(6, () => {
    value.addAssign(
      snoise(pos).mul(amplitude)
    );
    amplitude.mulAssign(0.5);
    pos.mulAssign(2.0);
  });
  return value;
});

// Conditionals
const colorFn = Fn(([n]) => {
  const col = vec3(0).toVar();
  If(n.greaterThan(0.5), () => {
    col.assign(vec3(1, 0, 0));
  }).Else(() => {
    col.assign(vec3(0, 0.3, 1));
  });
  return col;
});

// Chiamata — come JS normale
mat.colorNode = fbm(uv().mul(4));
✅ Fn() compila la funzione una volta nel GLSL generato — non inline. Codice pulito, riusabile.
⚠️ .toVar() obbligatorio per variabili mutabili — senza, il nodo è immutabile e assign fallisce silenziosamente.

// 09 — esempio reale
Esempio completo: Holographic Shader
holographic.js — tutto insieme
import * as THREE from 'three';
import { Fn, vec3, vec4, float, uniform,
  time, positionWorld, normalWorld,
  cameraPosition, sin, fract,
  dot, normalize,
  MeshBasicNodeMaterial } from 'three/tsl';

const uColor    = uniform(new THREE.Color('#00ffff'));
const uScanSpd  = uniform(2.0);
const uFresExp  = uniform(2.0);

const holoShader = Fn(() => {
  // 1. Scanlines animate
  const scan = fract(
    positionWorld.y.mul(20)
      .add(time.mul(uScanSpd))
  ).smoothstep(0.0, 0.3);

  // 2. Fresnel rim
  const vDir = normalize(
    cameraPosition.sub(positionWorld)
  );
  const fresnel = dot(normalWorld, vDir)
    .oneMinus().pow(uFresExp);

  // 3. Glitch
  const glitch = sin(
    positionWorld.y.mul(50)
      .add(time.mul(10))
  ).pow(20).mul(0.4);

  // Combina alpha layers
  const alpha = float(0.1)
    .add(scan.mul(0.4))
    .add(fresnel.mul(0.5))
    .add(glitch);

  return vec4(uColor, alpha);
});

const mat = new MeshBasicNodeMaterial({
  transparent: true,
  side: THREE.DoubleSide
});
mat.outputNode = holoShader();

// 10 — compute
Compute Shaders WebGPU only
Esegui simulazioni intere sulla GPU — 100k particelle aggiornate ogni frame senza toccare la CPU.
compute-particles.js
import { Fn, storage, instanceIndex,
  uniform, vec3, If,
  StorageBufferAttribute } from 'three/tsl';

const COUNT = 100_000;

// Buffer residenti sulla GPU
const posBuffer = new StorageBufferAttribute(COUNT, 3);
const velBuffer = new StorageBufferAttribute(COUNT, 3);

const posNode = storage(posBuffer, 'vec3', COUNT);
const velNode = storage(velBuffer, 'vec3', COUNT);
const uGravity = uniform(vec3(0, -9.8, 0));
const uDelta = uniform(0.016);

// Fn() che gira come compute shader
const updateFn = Fn(() => {
  const idx = instanceIndex;  // thread ID
  const pos = posNode.element(idx);
  const vel = velNode.element(idx);

  vel.addAssign(uGravity.mul(uDelta));
  pos.addAssign(vel.mul(uDelta));

  If(pos.y.lessThan(0), () => {
    pos.y.assign(0);
    vel.y.assign(
      vel.y.abs().mul(0.8)
    );
  });
});

// .compute(n) → dispatch n thread in parallelo
const computeNode = updateFn().compute(COUNT);

// render loop
await renderer.computeAsync(computeNode);

⚠️ I compute shader richiedono WebGPU. Fallback WebGL: usa GPGPU con render target (tecnica vecchia ma compatibile).

// 11 — best practices
Best Practices & Gotchas
✅ DO
.toVar() per variabili mutabili

const x = float(1).toVar() crea una variabile shader modificabile. Senza .toVar() il nodo è read-only. x.addAssign(1) funziona solo con .toVar().

Fn() per logica riutilizzabile

Wrappa la logica complessa in Fn(). Viene compilata come funzione separata — non inline. Più leggibile, riusabile tra materiali.

uniform() per dati dinamici

Qualsiasi valore che cambia ogni frame deve essere uniform(). Non creare nuovi nodi nel render loop — crea i nodi una volta, aggiorna solo .value.

❌ DON'T
Non mescolare operatori JS e TSL

float(1).add(2) ✅ — float(1) + 2 ❌. Gli operatori JavaScript (+ * ecc.) non sono overloadati in TSL. Usa sempre i metodi.

Non ricreare nodi ogni frame

Creare nodi TSL ha un costo. Il codice nella render loop deve solo aggiornare .value degli uniforms — non creare nuovi Fn(), vec3(), ecc.

Debug: leggi il GLSL generato

Usa renderer.debug.output = true per stampare il codice GLSL/WGSL generato. Molto utile per capire cosa sta succedendo davvero.

renderer-setup.js — fallback automatico


import { WebGLRenderer } from 'three';
import { WebGPURenderer } from
  'three/addons/renderers/WebGPURenderer';

let renderer;
if (await WebGPURenderer.isAvailable()) {
  renderer = new WebGPURenderer();
  renderer.backend = 'webgpu';
} else {
  // Fallback → GLSL automatico
  renderer = new WebGLRenderer();
}
// Lo stesso codice TSL funziona su entrambi!

      

      
⚠️ Safari: WebGPU in sviluppo (Safari TP). In produzione usa sempre il fallback WebGL. Il TSL gestisce la traduzione automaticamente.

      
✅ Per vertex/fragment shader normali (effetti, materiali personalizzati) TSL funziona identico su WebGL. Solo compute richiede WebGPU.
